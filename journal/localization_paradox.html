Localization Paradox Demo

A standalone walk-through showing how psm behaves as the memory backbone for the look-back queries described in recent work on the localization paradox in streaming egocentric memory — the gap between an MLLM’s semantic answer (“you saw a yellow bus”) and its near-zero temporal grounding (the [t_start, t_end] interval that proves it).

The premise of that line of work: when you ask a frontier MLLM a “where/when did I last see X?” question over hours of unscripted Ray-Ban-style video, it can usually generate a plausible-sounding answer, but its mIoU against a human-annotated visual proof interval is often ≈ 0. The model guesses from context and linguistic priors instead of pointing at an actual moment in the stream.

psm is structurally a candidate for closing that gap. Each H3 cell holds a ring buffer of HLL sketches with (t_min, t_max) retained per bucket and a per-tile reservoir of exemplar embeddings. That gives us, in O(matching_tiles × capacity):

  • --last-seen LAT,LNG — the most recent intervals where any observation landed in this neighborhood.
  • --search query.f32 — top-k tiles ranked by cosine similarity of their best stored exemplar against a CLIP text query embedding, each returning (cell, t_start, t_end, exemplar_t, similarity, count).

This document is a runnable end-to-end walk-through of those two query surfaces on the included Aria session.

What the demo shows

A single Aria session (one wearer, 15-min walk in Fulham/Sands End, GPS + IMU + 30 fps egocentric video subsampled to 3 fps for embeddings). For two natural-language cues — “a red bus”, “a zebra crossing” — psm --search returns ranked (H3 cell, t_start, t_end) tuples in well under a millisecond. Those intervals are then visually verified against the timeline by scrubbing psm-viz to each exemplar_t.

The point is not absolute retrieval accuracy on a benchmark. The point is the shape of the answer: PSM emits the kind of grounded interval the localization paradox literature explicitly says frontier MLLMs fail to produce on their own. PSM is best read as a prefilter — a candidate-interval generator that an MLLM consumes as grounding context, rather than a competitor to the MLLM.

Inputs

The session lives flat under datasets/1501677363692556/ with the Aria sidecar layout plus two HDF5 files:

datasets/1501677363692556/
├── data.mp4              # ~15 min, 30 fps egocentric video
├── gps.json              # 1 Hz GPS fixes
├── imu.json              # 100 Hz accel + gyro
├── metadata.json         # capture_time_epoch + session metadata
├── features.h5           # dino + jepa + gps + imu  (used by psm-viz)
└── clip_features.h5      # clip + gps + imu        (used by psm --search)

Pre-computed CLIP text-query embeddings live in the parent datasets/ directory:

  • datasets/q-bus.bin — CLIP text embedding for “a red bus”, float32 LE.
  • datasets/q-zebra.bin — CLIP text embedding for “a zebra crossing”, float32 LE.

These are exactly the format psm --search <path> expects.

Why two HDF5 files? The original Aria pipeline produced features.h5 with dino + jepa groups (used by psm-viz for attention/prediction overlays). clip is needed for the text-query path, but the in-tree extraction writes one group set per output file rather than appending. So we keep features.h5 as-is for the visualizer and write clip_features.h5 alongside it for the search path. A future add-group subcommand (TODO Phase 4) will let both groups live in one file.

Step 1 — Extract clip_features.h5

features.h5 ships with the session. clip_features.h5 does not — it must be extracted once. The in-tree extraction pipeline auto-detects the GPS/IMU sidecars next to the video, so the resulting file has real H3 cells along the captured route.

conda activate psm

# First-time only: install the optional CLIP runner.
pip install -e extraction[clip]

python -m psm_extraction extract \
  --video    datasets/1501677363692556/data.mp4 \
  --output   datasets/1501677363692556/clip_features.h5 \
  --models   clip \
  --sample-fps 3 --segment-sec 1 \
  --session-id 1501677363692556

Notes:

  • 25 seconds on M4 MPS for the included session (2707 frames × 512-d CLIP).
  • The --sample-fps 3 matches the existing dino group’s frame rate so timestamps line up across the two files.
  • A .clip_features.clip.<hash>.npz cache lands next to the output. Re-running picks up the cache; pass --force-reembed to skip it.
  • A future full re-extraction (--models clip,dino,jepa --sample-fps 30) takes 22 minutes on the same hardware and produces a single combined features.h5.

Step 2 — CLI look-back queries

Two retrieval surfaces matter for the localization paradox. Both already land in psm -j.

The defaults (-t 5.0 -C 12) only retain 60 s of memory per cell, which truncates everything outside the recent window in a 15-min session. For session-scale demos use -t 75 -C 12 (15 min) so older cells still report meaningful counts.

FEATURES=datasets/1501677363692556/clip_features.h5

2a. Visual-detail look-back: “where did I last see X?”

This is the closest analogue to the Location Trace and Visual Detail Recall categories in the benchmark.

# "a red bus"
targets/psm -f "$FEATURES" -g clip -t 75 -C 12 \
  --search datasets/q-bus.bin \
  --top 5 --exemplars 8 -j > /tmp/hits-bus.json

# "a zebra crossing"
targets/psm -f "$FEATURES" -g clip -t 75 -C 12 \
  --search datasets/q-zebra.bin \
  --top 5 --exemplars 8 -j > /tmp/hits-zebra.json

jq '.results[] | {cell, lat, lng, similarity, exemplar_t, t_min, t_max, count}' /tmp/hits-bus.json

Each entry in .results[] has similarity (cosine), exemplar_t (timestamp of the matching exemplar inside that tile), the bucket interval (t_min, t_max), and the HLL distinct-observation count over the active retention horizon. That set is the visual-proof grounding the localization paradox highlights as missing from raw MLLM responses.

Sample output from the included session (top-3 per query):

Query Cell (lat, lng) similarity exemplar_t (s) interval (s) count
q-bus.bin 8a194ada1a17fff (51.468216, -0.187567) 0.265 +258 [+251, +268] 55
q-bus.bin 8a194ada1a37fff (51.469027, -0.186435) 0.251 +321 [+269, +382] 342
q-bus.bin 8a194ada1baffff (51.470135, -0.187045) 0.248 +404 [+382, +467] 268
q-zebra.bin 8a194ada1a07fff (51.467918, -0.185826) 0.272 +242 [+161, +250] 268
q-zebra.bin 8a194ada1a37fff (51.469027, -0.186435) 0.264 +320 [+269, +382] 342
q-zebra.bin 8a194ada18c7fff (51.472945, -0.191748) 0.257 +781 [+705, +902] 598

Times are seconds-from-start (start ≈ Unix 1678365188.0). Cosine scores around 0.25–0.28 are typical for CLIP text↔image at base scale; the ranking matters more than the absolute number. The cell, (lat, lng), interval, and count columns are deterministic per-session; similarity and exemplar_t will drift slightly across runs because the per-tile exemplar reservoir is a sampler — different runs pick a different “best frame” inside the same cell.

2b. Place-anchored look-back: “what happened around here?”

Given a coordinate on the captured route, return recent intervals that touched it:

# Mid-route
targets/psm -f "$FEATURES" -g clip -t 75 -C 12 \
  --last-seen 51.470600,-0.188199 \
  --k-ring 1 --top 5 -j

Sample output:

{
  "results": [
    {"cell":"8a194ada1bb7fff","lat":51.471540,"lng":-0.189396,"t_min":1678365725.667,"t_max":1678365829.333,"count":309.359},
    {"cell":"8a194ada1b87fff","lat":51.470432,"lng":-0.188787,"t_min":1678365655.667,"t_max":1678365725.333,"count":215.087},
    {"cell":"8a194ada1baffff","lat":51.470135,"lng":-0.187045,"t_min":1678365570.000,"t_max":1678365655.333,"count":268.261}
  ]
}

Each result carries the H3 cell ID plus its center lat/lng so a human (or a script) can drop the coordinate into a map link without going back through H3.

--last-seen drops the embedding step entirely; it only needs the geographic prior. It is the spatial-side ablation that makes the embedding-driven retrieval in 2a meaningful: 2a should beat 2b on cue-driven queries (the cue narrows which cell among many that touch a coordinate); 2b alone can’t disambiguate “bus” from “crossing”.

Coordinates must be on the captured route. The session traces lat ∈ [51.4666, 51.4731], lng ∈ [-0.1925, -0.1855]. A coordinate outside that bounding box returns an empty results array (no H3 cells observed), not an error.

2c. New text queries

Generating a fresh query.f32 for an arbitrary phrase reuses scripts/e5_clip_demo.py, which embeds the text with the same CLIP head used for clip-group ingest:

python scripts/e5_clip_demo.py \
  datasets/1501677363692556/data.mp4 \
  "walking by the river" \
  --output-dir /tmp/psm-e5/river
# -> /tmp/psm-e5/river/query.f32 + psm_results.json

Drop query.f32 into Step 2a in place of q-bus.bin / q-zebra.bin and re-run.

Step 3 — Visual proof in psm-viz

For each top hit, scrub the visualizer to exemplar_t and confirm the cell is the right place at the right moment. This is the manual analogue of the benchmark’s “visual proof” interval requirement — the demo’s correctness check, not its evaluation harness.

targets/psm-viz -d datasets/1501677363692556 -g dino -m total

psm-viz reads features.h5 (with dino + jepa for overlays). The clip group used by --search lives in the separate clip_features.h5 and is not consumed by the visualizer.

For each (cell, exemplar_t) returned by Step 2a:

  1. Press Space to start playback (or scrub directly without playing).
  2. Scroll horizontally on the video pane to scrub to exemplar_t.
  3. Press M to cycle through total → current → recency heatmap modes — the cell should light up around exemplar_t and fade by t_max.
  4. Press E to toggle 3D hex extrusion if dominant cells are hard to read in the flat view.
  5. Press P to capture a screenshot into datasets/1501677363692556/captures/ for the demo writeup.

What to look for:

  • Under current, the queried cell visibly heats up only inside [t_min, t_max].
  • Under recency, the same cell stands out because the rest of the map has been visited more — i.e., the query landed somewhere genuinely novel for that route.
  • DINO attention overlay (inferno) on the video pane confirms the model was actually attending to the cued object at exemplar_t, not a confounder.

Step 3.5 — Embedding atlas (companion figure)

A geographic heatmap (Step 3) shows where memory landed. The scripts/embedding_atlas.py companion shows what memory contains — a 2D projection of the per-frame CLIP embeddings via UMAP (or PCA), with each cued query overlaid as a similarity gradient.

python scripts/embedding_atlas.py \
  datasets/1501677363692556/clip_features.h5 \
  --method umap \
  --query bus=datasets/q-bus.bin \
  --query zebra=datasets/q-zebra.bin \
  --query river=/tmp/psm-e5/river/query.f32 \
  --out captures/embedding_atlas_grid.png

How to read it:

  • Each dot is one video frame, positioned at the UMAP-projected coordinates of its 512-d CLIP embedding. Dots that are close in this view had visually similar content, regardless of when or where they were captured.
  • Color is cosine similarity between that frame and the panel’s text query. Computed in the original 512-d space, not the projected 2D.
  • White-edged circles mark the top-30 frames psm --search would return.

What the included session shows: all three queries (bus, zebra, river) light up roughly the same central blob; max similarities cluster around 0.27–0.29 within 0.01 of each other. That is negative evidence for the encoder, not for PSM. CLIP-base does not strongly discriminate fine-grained urban semantics on this footage; PSM faithfully retrieves whatever the encoder ranks highest. A stronger image encoder (CLIP-large, OpenCLIP-bigG, SigLIP-2) would tighten the panels visibly without any change to the engine.

The script also has a paired mode (--mode paired) that renders geographic and embedding panels side-by-side, both colored by the same H3 cell, to show the correspondence between spatial and semantic neighborhoods directly. See scripts/embedding_atlas.py --help.

Step 4 — Why this matters relative to the localization paradox

Spelled out plainly, ignoring metric numbers (which depend on the specific benchmark subset):

  • An MLLM asked “where did I last see the red bus?” and given only the live frame today cannot point at the historical interval — that is the paradox. The semantic head is fine; the temporal grounding head is not.
  • psm --search q-bus.bin returns (cell, t_start, t_end, exemplar_t) tuples in O(matching_tiles × capacity). On a session-scale memory that is sub-millisecond.
  • Forwarding those tuples plus the corresponding evidence frames to the MLLM as explicit grounding context turns “guess from priors” into “answer from a small set of candidate intervals.” That is a structural rebuttal to the paper’s discussion section, which calls for “adaptive temporal indexing… and hierarchical memory buffers that can compress hours of video without losing the granular detail of momentary ‘needle’ events.” psm’s H3-indexed ring-buffered HLL sketches plus per-tile exemplar reservoir are exactly that shape of data structure.

The demo is intentionally one session, two queries, one screenshot per hit. The full benchmark protocol — mIoU, R@1, GQ@0.5 over the eight cognitive categories — is documented in EXPERIMENTS.md (E5 / E6 / E7) and stays out of scope here.

Reproducibility checklist

Record alongside any captured artifacts:

  • git rev-parse HEAD
  • Build profile (local / portable / debug / sanitize)
  • Extraction command line and CLIP checkpoint string (default openai/clip-vit-base-patch32)
  • psm flags: --top, --exemplars, --k-ring, -r, -t, -C, -p, -g
  • For each top hit: query path, cell, similarity, exemplar_t, (t_min, t_max), screenshot filename

This lets a second pass (different checkpoint, different h3_resolution, different time_window_sec) be compared apples-to-apples against the original.
